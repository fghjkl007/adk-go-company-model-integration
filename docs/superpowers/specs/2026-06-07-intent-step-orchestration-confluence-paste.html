Purpose

Small demo design for showing how a Go Lambda can manage multi-step intents after Lex has already classified the first intent.

Key Idea

Lex classifies the entry intent.
LLM normalizes the user's answer into a DialogueCommand.
Go Orchestrator reads Lex SessionAttributes, validates/routes, updates ConversationState, and writes updated SessionAttributes.

Demo Scope

  • Entry intent comes from Lex.
  • Demo intents: MakePayment and LinkExternalAccount.
  • Intents are same-level; one intent does not own another.
  • MakePayment can route to LinkExternalAccount.
  • Resume happens only when a ResumeFrame exists.
  • On resume, return to the original intent and original step, then rerun Prepare -> Prompt.
  • Demo ConversationState is stored in Lex sessionAttributes.

Diagram 1: Runtime Architecture

#ComponentResponsibility
1UserUser says something in the voice flow.
2LexClassifies the entry intent and sends intent + utterance + session id to Lambda.
3Go LambdaReads Lex SessionAttributes.
4OrchestratorResolves active intent and active step.
5Current StepProvides the current ExpectedAnswerSpec.
6ADK / LLMReturns a normalized DialogueCommand.
7CommandGuardValidates command shape and canonical values against ExpectedAnswerSpec.
8OrchestratorRoutes the command to current step handling, delegate, resume, or reprompt.
9Step.HandleIf it is a current-step answer, runs business logic and returns StepResult.
10OrchestratorUpdates Lex SessionAttributes and returns the next assistant message.

Diagram 2: Step Lifecycle

#PhaseMeaning
1Enter stepThe flow enters or re-enters a step.
2PrepareAPI reads and business checks before asking the question.
3PromptBuild the assistant question and ExpectedAnswerSpec.
4WaitWait for user input.
5ADK / LLMReturn normalized DialogueCommand.
6CommandGuardValidate command shape and canonical value.
7Step.HandleFor answer / confirm / deny, run step business logic.
8Orchestrator routingFor change step or switch intent, route globally.
9RepromptFor unknown, reprompt or fallback.

Diagram 7: Command Routing

Command actMeaningOwner

Diagram 8: StepResult Apply Order

#ActionMeaning
1Update ConversationStateSet new values and clear stale values in the in-memory ConversationState.
2Apply transitionMove active step, delegate to another intent, or resume from ResumeFrame.
3Write SessionAttributesWrite the updated ConversationState back to Lex sessionAttributes.

Minimal Rules

TopicRule
Entry intentLex provides the first intent.
LLMLLM only returns structured DialogueCommand.
GuardCommandGuard validates command shape and canonical values against ExpectedAnswerSpec.
StepStep owns business logic and returns StepResult; it does not write Lex SessionAttributes.
OrchestratorOnly Orchestrator updates Lex SessionAttributes from StepResult and handles active step / delegate / resume.
StorageConversationState lives inside Lex sessionAttributes.
ResumeResume only if ResumeFrame exists.
Resume targetReturn to original intent + original step.
Resume executionRerun Prepare -> Prompt.

Demo Scenarios

  1. MakePayment happy path.
  2. MakePayment -> LinkExternalAccount -> resume MakePayment.
  3. Direct LinkExternalAccount completes without resuming MakePayment.
  4. User changes amount/date/account after later steps; disclosure becomes stale.
  5. User says automatic pay; LLM normalizes to autopay.
